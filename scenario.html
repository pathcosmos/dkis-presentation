2025.11.27 ① 착수 — "그냥 회귀 모델 하나 만들면 되는 거 아냐?" Week 0

사업 킥오프. 과제명은 "철강 압연공정 제조품질 관리 지능화". 핵심 목표는 2CRM 라인의 두께 실측값 예측. 초기 회의에서 가장 많이 나온 말은 "센서 데이터가 너무 많아서 다 쓸 수는 없을 것 같은데요"였고, 가장 많이 나온 반응은 "일단 다 넣고 돌려봅시다"였다.

초기 가정 ver21 노트북 파일 하나에 Pandas로 데이터 로드 → 피처 엔지니어링 → LightGBM 학습 → RMSE 보고 → 끝. 1주일이면 충분할 것 같았다.

초기 피처 후보 (너무 많았다)

  • CRM 센서 태그 (DK_2CR_*): 약 200개 후보
  • 기상 데이터: 8개 (TEMP, HUMI, WIND, ATM, SUN, SNOW, PRECIPITATION, GROUD_TEMP)
  • 시간/요일/주말: 10개 내외
  • 코일 메타 (RAWSPC_ID, PURCOM_ID, 강종): 50개 이상 (원핫 후)
"일단 전부 넣고 feature_importance 보고 고르죠" — 이 한마디가 앞으로 5개월을 결정했다.
2025.11.27 ~ 12.22 ② 데이터 지옥 — Machbase에서 데이터를 뺀다는 것 Week 1 ~ 4

처음 마주한 벽은 모델링이 아니라 데이터 소스였다. 센서 데이터는 Machbase에 있고, 코일 마스터는 Factory DB에, 기상은 기상청 API에. 세 개를 timestamp 기준으로 정렬해서 합치는 것부터가 프로젝트였다.

첫 번째 벽 Machbase WIDE_2CRM_THICK 테이블은 약 6.3억 건. 단일 쿼리로 3일치만 가져와도 Python이 메모리 터짐. Jupyter 노트북이 5분만에 커널 죽음. "이 데이터를 어떻게 모델링하라는 거지?"

시도한 것들 (전부 실패)

  • 시도 1: 전체 SELECT → 즉사 (OOM)
  • 시도 2: 시간 청크 단위 반복 조회 → Machbase 커넥터 타임아웃 빈발
  • 시도 3: Pandas read_sql chunksize → 메모리는 안 터지지만 12시간 이상 소요
  • 시도 4: 샘플링 → 정확도 떨어짐 + 재현성 문제
결단 팀장 한마디로 방향 전환: "Machbase → ClickHouse로 전부 이관합시다". 원래 사업 계획엔 없던 작업. 하지만 이 결정이 프로젝트를 살렸다.

ClickHouse 이관 (12/17 ~ 12/21)

  • WIDE_2CRM_THICK (6.3억 건): 2025-12-17 완료
  • WIDE_2CRM_OTHER (2,900만 건): 2025-12-21 완료
  • COIL_GRADE, Weather: 별도 이관 완료
결과 기존 Machbase에서 15분 걸리던 1일치 조회가 ClickHouse에서 2~3초. wide_2crm_thick → crm.crm_sensor_data. 이제서야 모델링을 시작할 수 있었다.
2025.11.27 ③ 첫 모델 ver22 — "왜 정확도가 이 모양이지?" Week 1 (병렬 진행)

데이터 이관과 병렬로, 샘플 데이터(1주일치)로 첫 모델 ver22를 만들었다. LightGBM 기본 파라미터. RMSE는… 크게 나왔다. 예측값이 실측값의 평균 근처를 왔다갔다.

ver22 초기
RMSE ≈ 수십~수백 μm
피처 200+, 기본 파라미터, 샘플 7일
→
ver22 최적화
RMSE 개선
파라미터 튜닝, OTIME 정렬, nan 제거

발견한 문제들

  • timestamp 미정렬: 센서 wide 테이블과 코일 마스터 조인 시 시간이 뒤섞여 있었음
  • NaN 폭탄: 센서 태그 중 일부는 90% 이상이 NaN
  • 강종(steel_grade) 누락: COIL_GRADE 매핑 실패 케이스 다수
  • 단위 불일치: 일부 태그는 μm, 일부는 mm. 섞여서 학습 중이었음
교훈 "Garbage In, Garbage Out"을 몸으로 배웠다. 모델 튜닝 전에 데이터 정제부터.
2025.11.27 ④ OOM 대참사 — 하루 종일 학습, 새벽에 커널 죽음 Week 1 후반

ver22를 전체 데이터로 돌리자 OOM (Out Of Memory). 64GB 서버에서도 죽었다. DataFrame 단일 로드 방식으로는 6.3억 건을 다룰 수 없었다.

# ver22 초기 — 이대로는 죽는다
df = pd.read_sql("SELECT * FROM crm.crm_sensor_data", conn)
# MemoryError: Unable to allocate 48.3 GiB for an array

해결 과정 — Parquet 청크 저장 방식 도입 (commit: "Fix OOM")

  • ClickHouse에서 월별 청크로 쿼리
  • 각 청크를 parquet로 즉시 저장 (RAM 해제)
  • 학습 시 parquet를 스트리밍 로드 → LightGBM의 native_format 이용
  • 중간 체크포인트 저장으로 재시작 가능하게
결과 메모리 피크 48GB → 8GB로 감소. 학습 시간은 여전히 6시간+지만, 적어도 완주 가능해짐.
"새벽 3시에 학습 돌려놓고 잠들었는데 아침에 커널이 죽어있던 날이 일주일 넘게 있었다. 결국 저장-재개 로직이 살렸다."
2025.11.28 ⑤ 피처 정리 ver23 — "중요도 0 피처가 6개?" Week 1 끝

ver23에서 stratified sampling + 피처 엔지니어링을 개선했다. feature_importance를 찍어보니 예상과 다른 결과가 나왔다.

68
학습 피처 수
6
중요도 0 피처
3
예상 외 0 피처
62
유효 피처

중요도 0의 미스터리

  • OTIME_YEAR, OTIME_MONTH: 데이터 기간이 좁아서 분산 없음 → 당연한 결과
  • SNOW: 2CRM 실내 공정. 예상 가능
  • DK_N2_CPU2_DB801_DD86: 예상 외. 알고 보니 전체 값이 상수였음. 센서 고장 가능성
  • DK_2CR_IBA_DB230_REAL76, REAL108, REAL232: 중요도 0. 수동 점검 결과 다른 핵심 피처와 완전 상관 (r=0.99+). LightGBM이 중복으로 판단하고 drop
발견 피처 중요도 0이 반드시 "쓸모없음"을 뜻하지는 않는다. "다른 피처와 겹침" 또는 "센서 이상"일 수 있다. 덕분에 역으로 고장난 센서를 역탐지하는 용도로 쓸 수 있다는 아이디어가 나왔다.
2025.12.23 ⑥ PSM 분기 — "강종이 다르면 모델도 달라야 하지 않나?" Week 4

ver23까지는 모든 강종을 하나의 모델로 학습했다. 그런데 잔차를 강종별로 쪼개서 봤더니… SK강에서 편향이 크고, SC강은 괜찮은데, 일반강은 표본이 너무 많아 모델을 잡아먹고 있었다.

문제 단일 모델이 "평균적으로" 잘 맞추지만, 특정 강종 + 특정 패스에서는 계통적으로 빗나감. 현장에서는 이게 바로 불량률로 이어진다.

PSM (Pass Schedule Management) 모델 탄생

두께 예측과 별개로 패스 스케줄 관리용 모델이 필요하다는 판단. 센서 태그가 아닌 공정 변수(두께/속도/장력/패스/강종)만으로 압연력을 예측하는 도메인 모델을 시도했다.

  • PSM v1: 강종 구분 없이 단일 모델 → 일반화 실패
  • PSM v2 (steel_grade 기반): 강종 문자열 매핑. 오타 섞인 케이스에서 터짐
  • PSM v3 (real_grade 기반): COIL_GRADE 연동, 강종별 독립 모델. 3개 강종(SC/일반/SK) 각각 학습
결론 PSM v3은 22 피처, 3개 강종별 모델로 확정. 메인 예측 모델(ver21)과는 역할이 다른 독립 모델로 유지하기로 결정. "한 모델로 모든 걸 풀려는 욕심을 내려놨다"는 게 큰 전환점.
2025.12.24 ⑦ 실시간 충격 — "13MB 모델이 8초 걸린다고요?" Week 5

realtime_app에 ver21 LightGBM 모델(13MB, .pkl)을 탑재해서 실시간 예측을 돌리려 했다. 웹소켓으로 센서 데이터를 받고, FastAPI 백엔드에서 예측을 반환하는 구조.

문제 ver21 모델 단일 추론에 약 200~500ms. 문제는 프로세스 시작 시 로딩 시간이 6~8초. systemd restart 할 때마다 "예측 서비스 죽었다" 알림이 날아왔다.

추가로 드러난 문제들

  • Python pickle 로드 + LightGBM 바인딩 초기화에만 5초 이상
  • 메모리 점유 약 600MB — 경량 서버에서 부담
  • 피처 준비 로직도 Pandas 기반이라 단건 추론에 불필요하게 무거움
  • feature 순서가 어긋나면 silent하게 이상한 예측 반환 (디버깅 악몽)
"배치에서는 최고의 모델이 실시간에서는 최악의 모델일 수 있다. 이걸 몸으로 배웠다."
2025.12.30 ⑧ 둘로 나누기 — ver21(배치) + v24 ONNX(실시간) Week 5~6 · 핵심 전환점

이 시점에 팀 전체가 모여 결정을 내렸다. "모델을 둘로 나누자". 하나는 정확도 우선의 배치 분석용 모델, 다른 하나는 속도 우선의 실시간 추론용 경량 모델.

설계 원칙

  • ver21 (배치): 68 피처 전부 유지. 정확도 최우선. 오프라인 분석과 리포트 생성용.
  • v24 ONNX (실시간): 피처 축소 + ONNX 변환 + 추론 최적화. 현장 모니터링용.

v24 피처 축소 작업 (68 → 50)

  • 중요도 0 피처 6개 제거
  • 상관 높은 피처 3쌍 중 하나씩 제거 (6개)
  • 실시간 Kafka 파이프라인에서 안정적으로 들어오지 않는 센서 6개 제거
  • 합계 18개 피처 제거 → 50개 확정

ONNX 변환 (2025.12.30 "ONNX v24 dual model support")

# LightGBM → ONNX 변환
from onnxmltools import convert_lightgbm
from skl2onnx.common.data_types import FloatTensorType
onnx_model = convert_lightgbm(lgbm, initial_types=[('input', FloatTensorType([None, 50]))])
이전 (ver21 단일)
13MB · 6~8초 로딩
단건 추론 200~500ms
→
이후 (v24 ONNX)
1.9MB · <500ms 로딩
단건 추론 5~15ms
효과 모델 크기 -85%, 추론 속도 ~20배 개선. 실시간 서비스가 드디어 가능해졌다. 배치 분석은 ver21을 그대로 사용 → 정확도 손실 없음.
대가 모델을 둘로 유지하는 비용: 학습 파이프라인 2개 유지, feature 정의 sync 필요, predictor 서비스 코드 분리 (prediction.py + prediction_onnx.py). "복잡도"가 붙었지만, 실시간 요구사항을 만족시킨 것이 더 중요했다.
2025.12.29 ⑨ 센서 매핑 지옥 — "예측값이 왜 목표치를 따라가?" Week 5

실시간 대시보드에 숫자가 흘러나오기 시작했는데, 예측값이 실측값이 아니라 목표치를 따라가는 기현상이 발생했다. "모델이 목표를 예측하고 있어요. 뭔가 이상해요."

원인 Kafka에서 들어오는 센서 태그를 v24 모델 feature와 매핑할 때, DB230_REAL4(실측 두께)와 DB230_REAL108(목표 두께)이 뒤바뀌어 있었음. 12/29 하루에만 5개의 매핑 버그 수정.

그 날 수정된 매핑 버그들 (commit 기록)

  • "Fix thickness data display by correcting sensor mapping" — 실측/목표 뒤바뀜
  • "Fix speed sensor mapping (REAL100/104 for entry/exit speed)" — 입측/출측 속도 반대
  • "Fix rolling status detection logic" — 압연 중/정지 상태 오탐
  • "Fix WebSocket broadcast and add Toast notifications" — 메시지 유실
  • "Improve data continuity and reduce gaps in realtime display" — 데이터 구멍
"12월 29일은 역사상 가장 긴 하루였다. 오전에 발견한 매핑 버그 하나가 새로운 버그 네 개를 끌고 나왔다."
교훈 센서 태그 명세서는 반드시 현장에서 검증해야 한다. IBA(PLC 로그) 채널 번호는 서류와 실제가 다른 경우가 존재한다. 그리고 모델은 잘못된 입력이 들어와도 소리 없이 틀린 답을 낸다.
2025.12.31 ~ 2026.04.10 ⑩ 마무리 — 배포, 모니터링, 최종 평가 Week 6 ~ 20

모델 두 개(ver21 + v24) + PSM v3이 자리를 잡은 이후는 주로 운영 안정화였다. 실시간 파이프라인을 Kafka 기반으로 이중화하고, 환경변수로 여러 환경을 지원하고, 성능 통계 서비스를 얹었다.

주요 마일스톤

  • 2025.12.31: ONNX v24 모델 컬럼을 realtime_sync 파이프라인에 통합
  • 2026.01.06: 2026년 데이터 테이블(THICK2, OTHER3)로 소스 전환
  • 2026.01.07: 기상 데이터 backfill 스크립트 완성
  • 2026.01.13: Multi-server 구성, systemd 서비스화, nginx 프록시
  • 2026.03.31: Kafka/ClickHouse 환경변수 지원 (remote 서버 대응)
  • 2026.04.09: 최종 평가 자료 배포
  • 2026.04.10: 피처 중요도 분석 리포트 추가
최종 상태 ver21 (배치) + v24 ONNX (실시간) + PSM v3 (도메인) 삼각구도로 운영. 메인 질의응답 페이지, 실시간 대시보드, 두께 예측 리포트 모두 안정 동작.

회고 — 다시 시작한다면

잘한 것

  • Machbase → ClickHouse 이관 결정: 없었으면 프로젝트 자체가 불가능
  • Parquet 청크 저장: OOM 없이 6.3억 건을 다룰 수 있게 된 결정적 기술
  • 모델 분리 (ver21 + v24): 배치와 실시간의 요구사항이 다름을 인정한 것
  • PSM을 별도 모델로 유지: "한 모델로 모든 걸" 욕심을 내려놓은 것

아쉬운 것

  • 센서 태그 검증을 미뤘던 것: 실시간 통합 단계에서야 매핑 오류 발견. 초기에 현장 검증이 있었다면 12/29 하루를 아낄 수 있었다.
  • 단위 통일을 나중에 한 것: μm/mm 혼재 상태로 며칠간 학습을 돌린 건 완전한 시간 낭비
  • 피처 중요도 0을 초기에 간과한 것: 센서 고장 힌트를 더 빨리 잡을 수 있었다
  • 버전 네이밍(ver21~ver24): 초기에 실험용/운영용 구분 없이 버전 숫자만 올리다 보니 어느 게 어디에 쓰이는지 혼동

다시 한다면 — 5가지 원칙

  1. 데이터 파이프라인부터 확정한다. 모델링은 그 다음. 데이터가 준비되지 않으면 아무것도 못한다.
  2. 배치와 실시간은 처음부터 다른 모델로 계획한다. 같이 가려다가 둘 다 망칠 수 있다.
  3. 센서 태그는 첫 주에 현장 검증한다. 서류와 실제는 다르다.
  4. 피처 중요도 0은 버그 신호일 수 있다. 버리기 전에 반드시 원인을 확인한다.
  5. 도메인 지식이 필요한 서브 문제는 별도 모델로 뽑는다. PSM처럼.
